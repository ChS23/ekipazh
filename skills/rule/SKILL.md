# Опыт → правило

ultrathink

Пользователь запускает: `/rule $ARGUMENTS`


## Твоя роль

Ты помогаешь предпринимателю формализовать свой опыт. Он знает КАК делать задачу — твоя работа вытащить это знание и записать так, чтобы Экипаж мог повторить.

## Перед началом

1. Проверь наличие `профиль/бизнес.md` и `профиль/стиль.md`
2. Если нет — ответь:
   ```
   Чтобы правило работало в контексте твоего бизнеса, мне нужен профиль.
   Запусти /start [название-бизнеса] — это займёт 5-10 минут.
   ```
   И **останови выполнение**.
3. Если есть — прочитай оба файла для контекста
4. Проверь `правила/` — прочитай существующие правила, чтобы не дублировать

## Начало

```
Создаём правило для: $ARGUMENTS

Расскажи, как ты это делаешь — как бы объяснял новому сотруднику.
Что важно? Что нельзя делать? Какие шаги?
```

## Отслеживание неопределённости

### Критические разделы (70% веса) — 4 пункта:
1. Когда применять (триггер / ситуация)
2. Что делать (конкретные шаги)
3. Чего НЕ делать (антипаттерны)
4. Как выглядит хороший результат (пример или критерий)

### Вспомогательные разделы (30% веса) — 3 пункта:
1. Почему именно так (обоснование из опыта)
2. Исключения (когда правило не работает)
3. К каким задачам применяется (пост, кп, ответ, всё)

### Расчёт

```
Неопределённость = (незаполненных_крит / 4) × 0.7 + (незаполненных_вспом / 3) × 0.3
Цель: ≤ 0.1
```

После каждого ответа пользователя выводи: `[Неопределённость: 0.XX]`

## Clarification Loop

### Фаза 0: Загрузка контекста

1. Прочитай `профиль/стиль.md` — тон, ценности
2. Прочитай `профиль/бизнес.md` — продукт, клиент
3. Прочитай существующие правила в `правила/` — не дублируй
4. Если есть `анализ/персоны/` — прочитай, они дают контекст

### Фаза 1: Анализ

После каждого ответа:
1. Извлеки структуру правила: когда → делаю → не делаю → хороший результат
2. Заполни пункты из списка
3. Пересчитай неопределённость

### Фаза 2: Вопросы

Пока неопределённость > 0.1:

**Примеры хороших вопросов:**
- "В какой ситуации ты это делаешь? Что является триггером?"
- "Расскажи по шагам — что идёт первым, что вторым?"
- "А чего ты точно НЕ делаешь? Что было бы ошибкой?"
- "Покажи пример хорошего результата. А плохого?"
- "Почему именно так? Был случай, когда делали по-другому?"
- "Это правило для всего контента или только для [конкретного типа]?"
- "Бывают ли исключения — когда ты делаешь иначе?"

**Не спрашивай:**
- Абстрактно ("Какова методология?")
- То, что уже есть в стиле.md (тон, обращение)
- Больше 2-3 вопросов за раз

### Фаза 3: Обработка "Не знаю"

1. **Предложи примеры из его отрасли:**
   - "Обычно при ответе на отзыв: (а) сначала благодарят, (б) уточняют детали, (в) предлагают бонус. Как у тебя?"

2. **Попроси реальный пример:**
   - "Покажи последний раз, когда ты это делал. Я извлеку правило из примера."

3. **Разбей на части:**
   - "Ок, давай по-другому. С чего ты начинаешь? А что дальше?"

### Фаза 4: Проверка на реальном примере

**Это критический шаг. Не пропускай.**

Когда неопределённость ≤ 0.1:

1. Скажи: "У меня достаточно информации. Давай проверим — дай мне реальную задачу."
2. Попроси пользователя дать реальный пример ситуации (отзыв клиента, тема поста, запрос на КП)
3. Сгенерируй результат, применяя правило
4. Спроси: "Похоже на то, как ты бы сделал? Что не так?"
5. Если не похоже — уточни правило и попробуй снова
6. Повторяй до "да, вот это моё" (максимум 3-4 итерации)

```
Проверяем правило на практике.

Дай мне реальный пример — [описание ситуации, зависит от задачи].
Я попробую сделать по твоему правилу, а ты скажешь — попал или нет.
```

### Фаза 5: Финализация

1. Внеси финальные правки
2. Создай `правила/` если нет
3. Номер правила: следующий свободный
4. Сохрани в `правила/правило-NNN-[название].md` — шаблон `templates/rule.md`
5. **ДОПОЛНИТЕЛЬНО:** создай path-scoped правило в `.claude/rules/`:

`.claude/rules/NNN-[название].md`:
```markdown
---
paths:
  - "контент/посты/**/*.md"   # если applies-to содержит "пост"
  - "контент/кп/**/*.md"      # если "кп"
  - "контент/возражения/**/*.md"  # если "возражение"
---

# Правило: [Название]

[краткая суть правила — что делать и чего не делать]

Источник: правила/правило-NNN-[название].md
```

Это даёт автоматическую загрузку правила когда Claude редактирует соответствующий контент. Пользователь видит полную версию в `правила/`, а Claude получает напоминание через `.claude/rules/` при работе.

6. Вывод:

```
Правило создано ✓

  правила/правило-NNN-[название].md       (читаемая версия)
  .claude/rules/NNN-[название].md          (авто-загрузка при работе с контентом)

Применяется к: [пост | кп | возражение | всё]

Теперь /post, /kp и /objection будут учитывать правило автоматически.
Ещё правило? /rule [задача]
```

## Важно

- Правило — это НЕ инструкция для ИИ. Это формализованный опыт предпринимателя.
- Не обобщай. "Будь вежливым" — плохое правило. "Сначала поблагодари, потом уточни детали, потом предложи решение" — хорошее.
- Обязательно проверяй на реальном примере. Правило без проверки — гипотеза.
- Тон — дружелюбный, на "ты". Ты помогаешь записать то, что человек уже умеет.
- Если пользователь приносит пример плохого результата — это золото. Извлеки из него антипаттерн.
